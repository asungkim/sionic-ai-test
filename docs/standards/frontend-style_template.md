# 프런트엔드 스타일 가이드

이 문서는 프런트엔드 개발 표준을 정의한다.
<!-- 프로젝트 스택에 맞게 전체적으로 수정하세요. 아래는 Next.js + React + TypeScript 기준 예시입니다. -->

## 1. 프로젝트 구조

- App Router 관례를 따른다.
- 기능 중심(feature-first) 레이아웃으로 컴포넌트/훅/테스트를 기능 디렉터리에 모은다.
- 파일명은 `kebab-case.tsx|ts`, 내보내는 React 컴포넌트는 PascalCase를 사용한다.

## 2. TypeScript 규칙

- `strict` 모드를 활성화한다.
- `any` 대신 `unknown`을 우선 사용하고, 묵시적 `any`를 허용하지 않는다.
- 상태 타입은 구분된 공용체(discriminated union)를 활용한다.
- 내보내는 함수에는 명시적 반환 타입을 작성한다.

## 3. React 규칙

- 함수형 컴포넌트 + 훅만 사용한다.
- 커스텀 훅 파일명은 `useXxx.ts`; 부수 효과 없는 순수/조합 가능한 훅을 지향한다.
- 렌더링 중 부수 효과를 피하고, `useEffect` 사용 시 정리(clean-up)를 반드시 구현한다.
- 성능 프로파일링 결과가 있을 때만 메모이제이션한다.
- 가능하면 서버 컴포넌트를 사용하고, 상호작용이 필요한 경우에만 클라이언트 컴포넌트를 사용한다.

## 4. 스타일링

- {{STYLING_APPROACH}} (예: Tailwind, CSS Modules, styled-components)를 우선 적용한다.
- 재사용 가능한 UI는 컴포넌트로 추출한다.
- 불필요한 `!important`나 임의 값 사용을 자제한다.

## 5. 데이터 및 API 레이어

- 가능한 경우 프레임워크 기본 데이터 패칭을 활용한다.
- 클라이언트 데이터 패칭은 훅으로 격리하고, 로딩/에러 상태를 일관되게 처리한다.
- API 타입은 자동 생성(OpenAPI 등)을 우선하고, 수동 타입 정의를 최소화한다.

## 6. 테스트

- 컴포넌트 테스트는 행동 기반 테스트를 우선한다 (예: React Testing Library).
- 스냅샷만의 테스트보다 사용자 인터랙션 기반 테스트를 우선한다.
- E2E 테스트는 주요 사용자 플로우를 커버한다 (예: Playwright, Cypress).

## 7. 포매팅 & 린팅

- {{LINTER}} (예: ESLint + typescript-eslint + 프레임워크 권장 설정)를 사용한다.
- {{FORMATTER}} (예: Prettier)로 포매팅한다.
- CI에서 린트/포맷 검사를 자동 실행한다.

## 8. 기타

- 에러 처리: 사용자 친화적 메시지를 우선하고, 기술적 세부사항은 개발 모드에서만 기록한다.
- 접근성: 시맨틱 HTML을 선호하고, 모달/동적 UI의 포커스 관리에 주의한다.
